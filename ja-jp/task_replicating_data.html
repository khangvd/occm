    システム間でデータを複製する GitHubで編集 ドキュメントの変更をリクエストする

    12/20/2019 寄稿者

    データ転送用に1回限りのデータ複製を選択するか、災害復旧または長期保存用の繰り返しスケジュールを選択することにより、作業環境間でデータを複製できます。たとえば、災害復旧のためにオンプレミスのONTAPシステムからCloud Volumes ONTAPへのデータ複製をセットアップできます。

    Cloud Managerは、SnapMirrorおよびSnapVaultテクノロジーを使用して、別々のシステム上のボリューム間のデータ複製を簡素化します。ソースボリュームとデスティネーションボリュームを特定し、レプリケーションポリシーとスケジュールを選択するだけです。 Cloud Managerは必要なディスクを購入し、関係を構成し、レプリケーションポリシーを適用してから、ボリューム間のベースライン転送を開始します。

    ベースライン転送には、ソースデータの完全なコピーが含まれます。後続の転送には、ソースデータの差分コピーが含まれます。

    データ複製の要件

    データを複製する前に、Cloud Volumes ONTAPシステムとONTAPクラスターの両方で特定の要件が満たされていることを確認する必要があります。

    バージョン要件

    データを複製する前に、ソースボリュームとデスティネーションボリュームが互換性のあるONTAPバージョンを実行していることを確認する必要があります。詳細については、 『 データ保護電源ガイド』を参照してください。

    Cloud Volumes ONTAPに固有の要件

    • インスタンスのセキュリティグループには、必要なインバウンドルールとアウトバウンドルールを含める必要があります。具体的には、ICMPとポート10000、11104、11105のルールです。

      これらのルールは、事前定義されたセキュリティグループに含まれています。

    • 異なるサブネットの2つのCloud Volumes ONTAPシステム間でデータを複製するには、サブネットを一緒にルーティングする必要があります（これはデフォルト設定です）。

    • AWSのCloud Volumes ONTAPシステムとAzureのシステム間でデータを複製するには、AWS VPCとAzure VNetの間にVPN接続が必要です。

    ONTAPクラスタに固有の要件

    • アクティブなSnapMirrorライセンスをインストールする必要があります。

    • クラスターが社内にある場合、企業ネットワークからAWSまたはAzureへの接続が必要です。これは通常、VPN接続です。

    • ONTAPクラスタは、追加のサブネット、ポート、ファイアウォール、およびクラスタ要件を満たす必要があります。

      詳細については、使用しているバージョンのONTAPのクラスターおよびSVMピアリングエクスプレスガイドを参照してください。

    システム間のデータ複製のセットアップ

    Cloud Volumes ONTAPシステムとONTAPクラスター間でデータをレプリケートするには、クラウドとの間でデータを移動するのに役立つワンタイムデータレプリケーションを選択するか、災害復旧または長期保存に役立つ定期的なスケジュールを選択します。

    このタスクについて

    Cloud Managerは、シンプル、ファンアウト、およびカスケードのデータ保護構成をサポートしています。

    • 単純な構成では、ボリュームAからボリュームBへの複製が行われます。

    • ファンアウト構成では、ボリュームAから複数の宛先に複製が行われます。

    • カスケード構成では、ボリュームAからボリュームBへ、およびボリュームBからボリュームCへの複製が行われます。

    システム間で複数のデータ複製を設定することにより、Cloud Managerでファンアウトおよびカスケード構成を構成できます。たとえば、システムAからシステムBにボリュームを複製してから、システムBからシステムCに同じボリュームを複製する

    手順

    1. [作業環境]ページで、ソースボリュームを含む作業環境を選択し、ボリュームをレプリケートする作業環境にドラッグします。

    2. [ソースおよび宛先ピアリング設定]ページが表示されたら、クラスターピア関係のクラスター間LIFをすべて選択します。

      クラスター間ネットワークは、クラスターピアがペアワイズフルメッシュ接続を持つように構成する必要があります。つまり、クラスターピア関係のクラスターの各ペアは、クラスター間LIFのすべての間で接続があります。

      これらのページは、複数のLIFを持つONTAPクラスターがソースまたはデスティネーションの場合に表示されます。

    3. [ソースボリュームの選択]ページで、複製するボリュームを選択します。

    4. [Destination Volume Name and Tiering]ページで、宛先ボリューム名を指定し、基盤となるディスクタイプを選択し、詳細オプションを変更して、[ 続行 ]をクリックします。

      デスティネーションがONTAPクラスタの場合、デスティネーションSVMとアグリゲートも指定する必要があります。

    5. [最大転送速度]ページで、データを転送できる最大速度（1秒あたりのメガバイト数）を指定します。

    6. [レプリケーションポリシー]ページで、既定のポリシーのいずれかを選択するか、[ 追加のポリシー ]をクリックして、高度なポリシーのいずれかを選択します。

      ヘルプについては、複製ポリシーの選択を参照してください。

      カスタムバックアップ（SnapVault）ポリシーを選択する場合、ポリシーに関連付けられたラベルは、ソースボリューム上のSnapshotコピーのラベルと一致する必要があります。詳細については、 バックアップポリシーの仕組みをご覧ください。

    7. [スケジュール]ページで、1回限りのコピーまたは定期的なスケジュールを選択します。

      いくつかのデフォルトのスケジュールが利用可能です。別のスケジュールが必要な場合は、System Managerを使用して宛先クラスターに新しいスケジュールを作成する必要があります。

    8. レビューページに、選択内容を確認し、[Go]をクリックします。

    結果

    Cloud Managerはデータ複製プロセスを開始します。 [レプリケーションステータス]ページでレプリケーションの詳細を表示できます。

    データ複製のスケジュールと関係の管理

    2つのシステム間でデータ複製をセットアップした後、Cloud Managerからデータ複製スケジュールと関係を管理できます。

    手順

    1. [作業環境]ページで、ワークスペース内のすべての作業環境または特定の作業環境のレプリケーションステータスを表示します。

      オプションアクション

      ワークスペース内のすべての作業環境

      Cloud Managerの上部で、[ レプリケーションステータス ]をクリックします。

      特定の作業環境

      作業環境を開き、[ レプリケーション ]をクリックします。

    2. データ複製関係の状態を調べて、それらが正常であることを確認します。

      関係のステータスがアイドルで、ミラー状態が初期化されていない場合、定義されたスケジュールに従ってデータ複製を行うには、宛先システムから関係を初期化する必要があります。 System Managerまたはコマンド行インターフェース（CLI）を使用して、関係を初期化できます。これらの状態は、宛先システムに障害が発生し、オンラインに戻ったときに表示されます。

    3. ソースボリュームの横にあるメニューアイコンを選択し、使用可能なアクションのいずれかを選択します。

      次の表に、使用可能なアクションを示します。

      アクション説明

      ブレーク

      ソースボリュームとデスティネーションボリュームの関係を解除し、データアクセスのためにデスティネーションボリュームをアクティブにします。

      通常、このオプションは、データの破損、誤った削除、オフライン状態などのイベントのためにソースボリュームがデータを提供できない場合に使用されます。

      データアクセス用のデスティネーションボリュームの設定とソースボリュームの再アクティブ化については、 『ONTAP 9 Volume Disaster Recovery Express Guide』を参照してください。

      再同期

      ボリューム間の壊れた関係を再確立し、定義されたスケジュールに従ってデータ複製を再開します。

      ボリュームを再同期すると、デスティネーションボリュームのコンテンツはソースボリュームのコンテンツによって上書きされます。

      デスティネーションボリュームからソースボリュームにデータを再同期する逆再同期を実行するには、 『 ONTAP 9 Volume Disaster Recovery Express Guide』を参照してください 。

      逆再同期

      ソースボリュームとデスティネーションボリュームの役割を逆にします。元のソースボリュームのコンテンツは、デスティネーションボリュームのコンテンツによって上書きされます。これは、オフラインになったソースボリュームを再アクティブ化する場合に役立ちます。

      最後のデータ複製とソースボリュームが無効にされた時間の間に元のソースボリュームに書き込まれたデータは保持されません。

      スケジュールを編集

      データ複製の別のスケジュールを選択できます。

      ポリシー情報

      データ複製関係に割り当てられた保護ポリシーを表示します。

      最大転送速度を編集

      データを転送できる最大速度（キロバイト/秒）を編集できます。

      更新

      増分転送を開始して、宛先ボリュームを更新します。

      削除

      ソースボリュームとデスティネーションボリュームの間のデータ保護関係を削除します。つまり、ボリューム間でデータ複製が行われなくなります。このアクションは、データアクセスの宛先ボリュームをアクティブにしません。また、このアクションは、システム間に他のデータ保護関係がない場合、クラスターピア関係とStorage Virtual Machine（SVM）ピア関係も削除します。

    結果

    アクションを選択すると、Cloud Managerは関係またはスケジュールを更新します。

    レプリケーションポリシーの選択

    Cloud Managerでデータ複製を設定するときに、複製ポリシーの選択に支援が必要になる場合があります。レプリケーションポリシーは、ストレージシステムがソースボリュームからデスティネーションボリュームにデータをレプリケートする方法を定義します。

    レプリケーションポリシーの機能

    ONTAPオペレーティングシステムは、Snapshotコピーと呼ばれるバックアップを自動的に作成します。 Snapshotコピーは、ある時点でのファイルシステムの状態をキャプチャするボリュームの読み取り専用イメージです。

    システム間でデータを複製する場合、Snapshotコピーをソースボリュームからデスティネーションボリュームに複製します。レプリケーションポリシーは、ソースボリュームからデスティネーションボリュームにレプリケートするSnapshotコピーを指定します。

    レプリケーションポリシーは、ディザスタリカバリ保護とディスク間バックアップおよびリカバリを提供するSnapMirrorおよびSnapVaultテクノロジーを使用しているため、 保護ポリシーとも呼ばれます。

    次の図は、Snapshotコピーとレプリケーションポリシーの関係を示しています。

    レプリケーションポリシーの種類

    レプリケーションポリシーには3つのタイプがあります。

    • ミラーポリシーは、新しく作成されたSnapshotコピーを宛先ボリュームに複製します。

      これらのSnapshotコピーを使用して、災害復旧または1回限りのデータ複製の準備としてソースボリュームを保護できます。データアクセスの宛先ボリュームをいつでもアクティブにできます。

    • バックアップポリシーは、特定のSnapshotコピーをデスティネーションボリュームにレプリケートし、通常、ソースボリューム上よりも長期間保持します。

      データが破損または失われた場合、これらのSnapshotコピーからデータを復元し、標準への準拠およびその他のガバナンス関連の目的のためにそれらを保持できます。

    • ミラーおよびバックアップポリシーは、災害復旧と長期保存の両方を提供します。

      各システムには、多くの状況で適切に機能するデフォルトのミラーおよびバックアップポリシーが含まれています。カスタムポリシーが必要な場合は、System Managerを使用して独自のポリシーを作成できます。

    次の図は、ミラーポリシーとバックアップポリシーの違いを示しています。ミラーポリシーは、ソースボリュームで使用可能なSnapshotコピーをミラーリングします。

    通常、バックアップポリシーは、ソースボリュームで保持されるよりも長くSnapshotコピーを保持します。

    バックアップポリシーの仕組み

    ミラーポリシーとは異なり、バックアップ（SnapVault）ポリシーは特定のSnapshotコピーを宛先ボリュームに複製します。デフォルトポリシーの代わりに独自のポリシーを使用する場合は、バックアップポリシーの仕組みを理解することが重要です。

    Snapshotコピーラベルとバックアップポリシーの関係について

    スナップショットポリシーは、システムがボリュームのスナップショットコピーを作成する方法を定義します。このポリシーは、Snapshotコピーをいつ作成するか、保持するコピーの数、およびそれらにラベルを付ける方法を指定します。たとえば、システムは、毎日午前12時10分に1つのSnapshotコピーを作成し、最新の2つのコピーを保持し、「毎日」というラベルを付けます。

    バックアップポリシーには、宛先ボリュームにレプリケートするラベル付きSnapshotコピーと保持するコピー数を指定するルールが含まれます。バックアップポリシーで定義されたラベルは、スナップショットポリシーで定義された1つ以上のラベルと一致する必要があります。そうしないと、システムはSnapshotコピーを複製できません。

    たとえば、ラベル「daily」および「weekly」を含むバックアップポリシーでは、それらのラベルのみを含むSnapshotコピーが複製されます。次の図に示すように、他のSnapshotコピーは複製されません。

    デフォルトポリシーとカスタムポリシー

    デフォルトのSnapshotポリシーは、1時間ごと、1日ごと、1週間ごとのSnapshotコピーを作成し、6時間ごと、2日ごと、および2週間ごとのSnapshotコピーを保持します。

    デフォルトのバックアップポリシーとデフォルトのスナップショットポリシーを簡単に使用できます。デフォルトのバックアップポリシーは、毎日および毎週のSnapshotコピーを複製し、毎日7つのスナップショットコピーと52の毎週のSnapshotコピーを保持します。

    カスタムポリシーを作成する場合、それらのポリシーで定義されたラベルが一致する必要があります。 System Managerを使用してカスタムポリシーを作成できます。
