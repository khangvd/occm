    データ階層化の概要 GitHubで編集 ドキュメントの変更をリクエストする

    02/03/2020 寄稿者

    非アクティブなデータを低コストのオブジェクトストレージに自動階層化することにより、ストレージコストを削減します。アクティブデータは高性能SSDまたはHDDに残り、非アクティブデータは低コストのオブジェクトストレージに階層化されます。これにより、プライマリストレージのスペースを再利用し、セカンダリストレージを縮小できます。

    Cloud Volumes ONTAPは、AWS、Azure、およびGoogle Cloud Platformでのデータ階層化をサポートしています。データ階層化は、FabricPoolテクノロジーによって強化されています。

    データの階層化（FabricPool）を有効にするために機能ライセンスをインストールする必要はありません。

    AWSのデータ階層化

    AWSでデータ階層化を有効にすると、Cloud Volumes ONTAPはEBSをホットデータのパフォーマンス層として使用し、AWS S3を非アクティブデータの容量層として使用します。

    パフォーマンス層

    パフォーマンス層は、汎用SSD、プロビジョニングされたIOPS SSD、またはスループットが最適化されたHDDです。

    容量ティア

    Cloud Volumes ONTAPシステムは、 標準ストレージクラスを使用して、非アクティブなデータを単一のS3バケットに階層化します。標準は、複数のアベイラビリティーゾーンに保存され、頻繁にアクセスされるデータに最適です。

    Cloud Managerは、作業環境ごとに単一のS3バケットを作成し、それにfabric-pool- cluster unique identifierという名前を付けます 。ボリュームごとに異なるS3バケットは作成されません。
    ストレージクラス

    AWSの階層データのデフォルトのストレージクラスはStandardです。非アクティブなデータにアクセスする予定がない場合、ストレージクラスを次のいずれかに変更することにより、ストレージコストを削減できます。 インテリジェントティアリング 、 1ゾーンの低頻度アクセス 、または標準の低頻度アクセス 。ストレージクラスを変更すると、非アクティブなデータは標準ストレージクラスで開始され、データが30日後にアクセスされない場合は、選択したストレージクラスに移動します。

    データにアクセスする場合、アクセスコストが高くなるため、ストレージクラスを変更する前にそれを考慮してください。 Amazon S3ストレージクラスの詳細 。

    作業環境を作成するときにストレージクラスを選択でき、その後いつでも変更できます。ストレージクラスの変更の詳細については、 非アクティブなデータを低コストのオブジェクトストレージに階層化するを参照してください。

    データ階層化のストレージクラスはシステム全体であり、ボリュームごとではありません。

    Azureのデータ階層化

    Azureでデータ階層化を有効にすると、Cloud Volumes ONTAPは、ホットデータのパフォーマンス層としてAzure管理対象ディスクを使用し、非アクティブデータの容量層としてAzure Blobストレージを使用します。

    パフォーマンス層

    パフォーマンス層はSSDまたはHDDのいずれかです。

    容量ティア

    Cloud Volumes ONTAPシステムは、Azure ホットストレージ層を使用して、非アクティブなデータを単一のBlobコンテナーに階層化します。ホットティアは、頻繁にアクセスされるデータに最適です。

    Cloud Managerは、Cloud Volumes ONTAPの作業環境ごとに1つのコンテナを持つ新しいストレージアカウントを作成します。ストレージアカウントの名前はランダムです。ボリュームごとに異なるコンテナは作成されません。
    ストレージアクセス層

    Azureの階層データのデフォルトのストレージアクセス層は、 ホット層です。非アクティブなデータにアクセスする予定がない場合は、 クールストレージ層に変更することでストレージコストを削減できます。ストレージ層を変更すると、非アクティブなデータはホットストレージ層で開始され、30日後にデータにアクセスできない場合はクールストレージ層に移動します。

    データにアクセスする場合、アクセスコストが高くなるため、ストレージ層を変更する前にそれを考慮してください。 Azure Blobストレージアクセス層の詳細 。

    作業環境を作成するときにストレージ層を選択でき、その後いつでも変更できます。ストレージ階層の変更の詳細については、 非アクティブなデータを低コストのオブジェクトストレージに階層化するを参照してください。

    データ階層化のストレージアクセス層はシステム全体であり、ボリュームごとではありません。

    GCPのデータ階層化

    GCPでデータ階層化を有効にすると、Cloud Volumes ONTAPは、ホットデータのパフォーマンス層として永続ディスクを使用し、非アクティブデータの容量層としてGoogle Cloud Storageバケットを使用します。

    パフォーマンス層

    パフォーマンス層は、SSDまたはHDD（標準ディスク）のいずれかです。

    容量ティア

    Cloud Volumes ONTAPシステムは、 リージョナルストレージクラスを使用して、非アクティブなデータを単一のGoogle Cloud Storageバケットに階層化します。

    Cloud Managerは、作業環境ごとに1つのバケットを作成し、fabric-pool- cluster unique identifierという名前を付けます 。ボリュームごとに異なるバケットは作成されません。
    ストレージクラス

    階層化データのデフォルトのストレージクラスは、 Standard Storageクラスです。データへのアクセス頻度が低い場合は、 Nearline StorageまたはColdline Storageに変更することにより、ストレージコストを削減できます。ストレージクラスを変更すると、非アクティブなデータは標準ストレージクラスで開始され、データが30日後にアクセスされない場合は、選択したストレージクラスに移動します。

    データにアクセスする場合、アクセスコストが高くなるため、ストレージクラスを変更する前にそれを考慮してください。 Google Cloud Storageのストレージクラスの詳細をご覧ください 。

    作業環境を作成するときにストレージ層を選択でき、その後いつでも変更できます。ストレージクラスの変更の詳細については、 非アクティブなデータを低コストのオブジェクトストレージに階層化するを参照してください。

    データ階層化のストレージクラスはシステム全体であり、ボリュームごとではありません。

    データの階層化と容量の制限

    データの階層化を有効にしても、システムの容量制限は変わりません。制限は、パフォーマンス層と容量層全体に広がります。

    ボリューム階層化ポリシー

    データ階層化を有効にするには、ボリュームを作成、変更、または複製するときにボリューム階層化ポリシーを選択する必要があります。ボリュームごとに異なるポリシーを選択できます。

    一部の階層化ポリシーには、最小冷却期間が関連付けられており、これにより、データが「低温」と見なされて容量層に移動するために、ボリューム内のユーザーデータが非アクティブのままである時間が設定されます。

    Cloud Managerでは、ボリュームを作成または変更するときに、次のボリューム階層化ポリシーから選択できます。

    スナップショットのみ

    アグリゲートの容量が50％に達すると、Cloud Volumes ONTAP層は、アクティブなファイルシステムに関連付けられていないSnapshotコピーのユーザーデータを容量層にコールドします。冷却期間は約2日間です。

    読み取られた場合、容量層のコールドデータブロックはホットになり、パフォーマンス層に移動されます。

    オート

    アグリゲートの容量が50％に達すると、Cloud Volumes ONTAPはボリューム内のコールドデータブロックを容量層に階層化します。コールドデータには、Snapshotコピーだけでなく、アクティブなファイルシステムからのコールドユーザーデータも含まれます。冷却期間は約31日間です。

    このポリシーは、Cloud Volumes ONTAP 9.4以降でサポートされています。

    ランダム読み取りで読み取られた場合、容量層のコールドデータブロックはホットになり、パフォーマンス層に移動します。インデックススキャンやアンチウイルススキャンに関連するものなど、シーケンシャルリードで読み取る場合、コールドデータブロックはコールドのままで、パフォーマンス層に移動しません。

    なし

    ボリュームのデータをパフォーマンス層に保持し、それが容量層に移動するのを防ぎます。

    ボリュームを複製するときに、データをオブジェクトストレージに階層化するかどうかを選択できます。その場合、Cloud Managerはバックアップポリシーをデータ保護ボリュームに適用します。 Cloud Volumes ONTAP 9.6以降、 すべての階層化ポリシーがバックアップポリシーを置き換えます。

    Cloud Volumes ONTAPをオフにすると、冷却期間に影響します

    データブロックは、冷却スキャンによって冷却されます。このプロセス中、使用されていないブロックのブロック温度は、次に低い値に移動（冷却）されます。デフォルトの冷却時間は、ボリューム階層化ポリシーによって異なります。

    • 自動：31日

    • スナップショットのみ：2日間

    冷却スキャンを機能させるには、Cloud Volumes ONTAPが実行されている必要があります。 Cloud Volumes ONTAPをオフにすると、冷却も停止します。その結果、冷却時間が長くなる可能性があります。

    データ階層化のセットアップ

    手順とサポートされる構成のリストについては、 非アクティブなデータを低コストのオブジェクトストレージに階層化するを参照してください。
