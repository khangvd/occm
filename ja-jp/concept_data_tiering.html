    データ階層化の概要 GitHubで編集 ドキュメントの変更をリクエストする

    06/18/2019 寄稿者

    低コストのオブジェクトストレージへの非アクティブデータの自動階層化を有効にして、ストレージコストを削減します。アクティブデータは高性能SSDまたはHDDに残り、非アクティブデータは低コストのオブジェクトストレージに階層化されます。これにより、プライマリストレージのスペースを再利用し、セカンダリストレージを縮小できます。

    Cloud Volumes ONTAPは、AWS、Azure、およびGoogle Cloud Platformでのデータ階層化をサポートしています。データ階層化は、FabricPoolテクノロジーによって強化されています。

    データ階層化（FabricPool）を有効にするために機能ライセンスをインストールする必要はありません。

    AWSのデータ階層化

    AWSでデータ階層化を有効にすると、Cloud Volumes ONTAPはホットデータのパフォーマンス層としてEBSを使用し、非アクティブデータの容量層としてAWS S3を使用します。システムの階層化レベルを変更すると、異なるS3ストレージクラスを選択できます。

    パフォーマンス層

    パフォーマンス層は、汎用SSD、プロビジョニングされたIOPS SSD、またはスループットが最適化されたHDDです。

    容量ティア

    Cloud Volumes ONTAPシステムは、 標準ストレージクラスを使用して、非アクティブなデータを単一のS3バケットに階層化します。標準は、複数のアベイラビリティーゾーンに保存されている頻繁にアクセスされるデータに最適です。

    Cloud Managerは、作業環境ごとに単一のS3バケットを作成し、fabric-pool- cluster unique identifierという名前を付けます 。ボリュームごとに異なるS3バケットは作成されません。
    階層化レベル

    非アクティブなデータにアクセスする予定がない場合は、システムの階層化レベルを次のいずれかに変更することにより、ストレージコストを削減できます。 インテリジェント階層化、 ワンゾーン低頻度アクセス 、または標準低頻度アクセス 。階層化レベルを変更すると、非アクティブなデータは標準ストレージクラスで開始され、データが30日後にアクセスされない場合、選択したストレージクラスに移動します。

    データにアクセスする場合、アクセスコストが高くなるため、階層化レベルを変更する前にそれを考慮してください。 Amazon S3ストレージクラスの詳細 。

    システムを作成した後、階層レベルを変更することができます。詳細については、 非アクティブなデータを低コストのオブジェクトストレージに階層化するを参照してください。

    階層化レベルはシステム全体です。ボリュームごとではありません。

    Azureのデータ階層化

    Azureでデータ階層化を有効にすると、Cloud Volumes ONTAPは、ホットデータのパフォーマンス層としてAzure管理対象ディスクを使用し、非アクティブデータの容量層としてAzure Blobストレージを使用します。システムの階層化レベルを変更すると、別のAzureストレージ階層を選択できます。

    パフォーマンス層

    パフォーマンス層はSSDまたはHDDのいずれかです。

    容量ティア

    Cloud Volumes ONTAPシステムは、Azure ホットストレージ層を使用して、非アクティブなデータを単一のBlobコンテナーに階層化します。ホットティアは、頻繁にアクセスされるデータに最適です。

    Cloud Managerは、Cloud Volumes ONTAPの作業環境ごとに1つのコンテナを持つ新しいストレージアカウントを作成します。ストレージアカウントの名前はランダムです。ボリュームごとに異なるコンテナは作成されません。
    階層化レベル

    非アクティブなデータにアクセスする予定がない場合は、システムの階層化レベルをAzureのクールなストレージ階層に変更することにより、ストレージコストを削減できます。階層化レベルを変更すると、非アクティブなデータはホットストレージ層で開始され、30日後にデータにアクセスできない場合はクールストレージ層に移動します。

    データにアクセスする場合、アクセスコストが高くなるため、階層化レベルを変更する前にそれを考慮してください。 Azure Blobストレージアクセス層の詳細 。

    システムを作成した後、階層レベルを変更することができます。詳細については、 非アクティブなデータを低コストのオブジェクトストレージに階層化するを参照してください。

    階層化レベルはシステム全体です。ボリュームごとではありません。

    GCPのデータ階層化

    GCPでデータ階層化を有効にすると、Cloud Volumes ONTAPは、ホットデータのパフォーマンス層として永続ディスクを使用し、非アクティブデータの容量層としてGoogle Cloud Storageバケットを使用します。

    パフォーマンス層

    パフォーマンス層は、SSDまたはHDD（標準ディスク）のいずれかです。

    容量ティア

    Cloud Volumes ONTAPシステムは、 リージョナルストレージクラスを使用して、非アクティブなデータを単一のGoogle Cloud Storageバケットに階層化します。

    Cloud Managerは、作業環境ごとに1つのバケットを作成し、fabric-pool- cluster unique identifierという名前を付けます 。ボリュームごとに異なるバケットは作成されません。
    階層化レベル

    現時点では、他のGCPストレージクラスはサポートされていません。

    データの階層化と容量の制限

    データ階層化を有効にしても、システムの容量制限は変わりません。制限は、パフォーマンス層とキャパシティー層全体に広がります。

    ボリューム階層化ポリシー

    データ階層化を有効にするには、ボリュームを作成、変更、または複製するときにボリューム階層化ポリシーを選択する必要があります。ボリュームごとに異なるポリシーを選択できます。

    一部の階層化ポリシーには、最小冷却期間が関連付けられています。これにより、データが「コールド」と見なされて容量層に移動されるまで、ボリューム内のユーザーデータが非アクティブのままである時間が設定されます。

    Cloud Managerでは、ボリュームを作成または変更するときに、次のボリューム階層化ポリシーから選択できます。

    スナップショットのみ

    アグリゲートが50％の容量に達すると、Cloud Volumes ONTAP層は、アクティブなファイルシステムに関連付けられていないSnapshotコピーのユーザーデータを容量層にコールドします。冷却期間は約2日間です。

    読み取られると、容量層のコールドデータブロックがホットになり、パフォーマンス層に移動されます。

    オート

    アグリゲートの容量が50％に達すると、Cloud Volumes ONTAPはボリューム内のコールドデータブロックを容量階層に階層化します。コールドデータには、Snapshotコピーだけでなく、アクティブファイルシステムからのコールドユーザーデータも含まれます。冷却期間は約31日間です。

    このポリシーは、Cloud Volumes ONTAP 9.4以降でサポートされています。

    ランダム読み取りで読み取られた場合、容量層のコールドデータブロックはホットになり、パフォーマンス層に移動します。インデックススキャンやアンチウイルススキャンに関連するものなど、シーケンシャルリードで読み取る場合、コールドデータブロックはコールドのままで、パフォーマンス層に移動しません。

    無し

    ボリュームのデータをパフォーマンス層に保持し、容量層に移動できないようにします。

    ボリュームを複製するときに、DRデータをオブジェクトストレージに階層化するかどうかを選択できます。その場合、Cloud Managerはバックアップポリシーをデータ保護ボリュームに適用します。 Cloud Volumes ONTAP 9.6以降では、 すべての階層化ポリシーがバックアップポリシーを置き換えます。

    データ階層化のセットアップ

    手順とサポートされる構成のリストについては、 非アクティブなデータを低コストのオブジェクトストレージに階層化するを参照してください。
