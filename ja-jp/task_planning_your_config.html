    Cloud Volumes ONTAP構成の計画 GitHubで編集 ドキュメントの変更をリクエストする

    12/16/2019 寄稿者

    Cloud Volumes ONTAPを展開するとき、ワークロード要件に一致する事前構成済みシステムを選択するか、独自の構成を作成できます。独自の構成を選択する場合は、使用可能なオプションを理解する必要があります。

    ライセンスの種類を選択する

    Cloud Volumes ONTAPには、従量課金制と独自のライセンス（BYOL）の2つの価格設定オプションがあります。従量課金の場合、Explore、Standard、Premiumの3つのライセンスから選択できます。各ライセンスは、異なる容量と計算オプションを提供します。

    • AWSのCloud Volumes ONTAP 9.7でサポートされている構成

    • AzureのCloud Volumes ONTAP 9.7でサポートされている構成

    • GCPのCloud Volumes ONTAP 9.7でサポートされる構成

    ストレージ制限について

    Cloud Volumes ONTAPシステムのraw容量制限は、ライセンスに関連付けられています。追加の制限は、アグリゲートとボリュームのサイズに影響します。構成を計画するときは、これらの制限に注意する必要があります。

    • AWSでのCloud Volumes ONTAP 9.7のストレージ制限

    • AzureのCloud Volumes ONTAP 9.7のストレージ制限

    • GCPのCloud Volumes ONTAP 9.7のストレージ制限

    書き込み速度を選択する

    Cloud Managerを使用すると、単一ノードのCloud Volumes ONTAPシステムの書き込み速度設定を選択できます。書き込み速度を選択する前に、通常の設定と高い設定の違い、および高い書き込み速度を使用する場合のリスクと推奨事項を理解する必要があります。

    通常の書き込み速度と高速の書き込み速度の違い

    通常の書き込み速度を選択すると、データが直接ディスクに書き込まれるため、計画外のシステム停止が発生した場合のデータ損失の可能性が低くなります。

    高い書き込み速度を選択すると、データはディスクに書き込まれる前にメモリにバッファリングされるため、書き込みパフォーマンスが向上します。このキャッシュにより、計画外のシステム停止が発生した場合、データ損失の可能性があります。

    計画外のシステム停止が発生した場合に失われる可能性のあるデータの量は、最後の2つの一貫性ポイントのスパンです。一貫性ポイントは、バッファリングされたデータをディスクに書き込む行為です。整合性ポイントは、書き込みログがいっぱいになるか、10秒後に（どちらか早い方）発生します。ただし、AWS EBSボリュームのパフォーマンスは、一貫性ポイントの処理時間に影響する可能性があります。

    高い書き込み速度を使用する場合

    ワークロードに高速の書き込みパフォーマンスが必要で、計画外のシステム停止が発生した場合のデータ損失のリスクに耐えることができる場合は、高速の書き込み速度が適しています。

    高い書き込み速度を使用する場合の推奨事項

    高速の書き込み速度を有効にする場合は、アプリケーション層で書き込み保護を確保する必要があります。

    ボリューム使用プロファイルの選択

    ONTAPには、必要なストレージの合計量を削減できるストレージ効率機能がいくつか含まれています。 Cloud Managerでボリュームを作成するとき、これらの機能を有効にするプロファイルまたは無効にするプロファイルを選択できます。使用するプロファイルを決定するのに役立つこれらの機能の詳細を学習する必要があります。

    NetAppストレージ効率化機能には、次の利点があります。

    シンプロビジョニング

    物理ストレージプールに実際にあるよりも多くの論理ストレージをホストまたはユーザーに提示します。ストレージスペースを事前に割り当てる代わりに、データの書き込み時にストレージスペースが各ボリュームに動的に割り当てられます。

    重複排除

    同一のデータブロックを見つけ、それらを単一の共有ブロックへの参照に置き換えることにより、効率が向上します。この手法は、同じボリュームに存在するデータの冗長ブロックを排除することにより、ストレージ容量の要件を削減します。

    圧縮

    プライマリ、セカンダリ、およびアーカイブストレージのボリューム内のデータを圧縮することにより、データの保存に必要な物理容量を削減します。

    AWSの計画

    システムのサイズを変更し、入力する必要があるネットワーク情報を確認して、AWSでのCloud Volumes ONTAPの展開を計画します。

    • AWSでシステムをサイジングする

    • AWSネットワーク情報ワークシート

    AWSでシステムをサイジングする

    Cloud Volumes ONTAPシステムのサイジングは、パフォーマンスと容量の要件を満たすのに役立ちます。インスタンスタイプ、ディスクタイプ、ディスクサイズを選択する際には、いくつかの重要な点に注意する必要があります。

    インスタンスタイプ

    • ワークロード要件を、各EC2インスタンスタイプの最大スループットとIOPSに一致させます。

    • 複数のユーザーが同時にシステムに書き込む場合、リクエストを管理するのに十分なCPUを備えたインスタンスタイプを選択します。

    • ほとんどが読み取りであるアプリケーションがある場合は、十分なRAMを備えたシステムを選択します。

      • AWSドキュメント：Amazon EC2インスタンスタイプ

      • AWSドキュメント：Amazon EBS –最適化されたインスタンス

    EBSディスクタイプ

    汎用SSDは、Cloud Volumes ONTAPで最も一般的なディスクタイプです。 EBSディスクのユースケースを表示するには、 AWSドキュメント：EBS Volume Typesを参照してください。

    EBSディスクサイズ

    Cloud Volumes ONTAPシステムを起動するときに、初期ディスクサイズを選択する必要があります。その後、 Cloud Managerにシステムの容量を管理させることができますが、 アグリゲートを自分で構築する場合は、次のことに注意してください 。

    • アグリゲート内のすべてのディスクは同じサイズでなければなりません。

    • EBSディスクのパフォーマンスは、ディスクサイズに関係しています。サイズにより、SSDディスクのベースラインIOPSと最大バースト期間、およびHDDディスクのベースラインとバーストスループットが決まります。

    • 最終的には、必要な持続的なパフォーマンスが得られるディスクサイズを選択する必要があります。

    • より大きなディスク（6つの4 TBディスクなど）を選択した場合でも、EC2インスタンスが帯域幅制限に達する可能性があるため、すべてのIOPSを取得できない場合があります。

      EBSディスクのパフォーマンスの詳細については、 AWSドキュメント：EBS Volume Typesを参照してください。

    AWSでのCloud Volumes ONTAPシステムのサイズ設定の詳細については、次のビデオをご覧ください。

    Azureプランニング

    システムのサイズを変更し、入力する必要があるネットワーク情報を確認して、AzureでのCloud Volumes ONTAPの展開を計画します。

    • Azureでのシステムのサイズ変更

    Azureでのシステムのサイズ変更

    Cloud Volumes ONTAPシステムのサイジングは、パフォーマンスと容量の要件を満たすのに役立ちます。 VMタイプ、ディスクタイプ、およびディスクサイズを選択する際には、いくつかの重要な点に注意する必要があります。

    仮想マシンのタイプ

    Cloud Volumes ONTAPリリースノートでサポートされている仮想マシンタイプを確認し、サポートされている各VMタイプの詳細を確認します。各VMタイプは特定の数のデータディスクをサポートすることに注意してください。

    • Azureドキュメント：汎用仮想マシンのサイズ

    • Azureドキュメント：メモリ最適化された仮想マシンのサイズ

    Azureディスクタイプ

    Cloud Volumes ONTAPのボリュームを作成する場合、Cloud Volumes ONTAPがディスクとして使用する基盤となるクラウドストレージを選択する必要があります。

    HAシステムはプレミアムページBLOBを使用します。一方、シングルノードシステムでは、2種類のAzure Managed Disksを使用できます。

    • Premium SSD Managed Disksは、I / O集中型のワークロードに対して高いコストで高いパフォーマンスを提供します。

    • 標準のSSD管理ディスクは、低いIOPSを必要とするワークロードに一貫したパフォーマンスを提供します。

    • 標準のHDD管理ディスクは、高いIOPSを必要とせず、コストを削減したい場合に適しています。

      これらのディスクの使用例の詳細については、 Microsoft Azure Documentation：Introduction to Microsoft Azure Storageを参照してください。

    Azureディスクサイズ

    Cloud Volumes ONTAPインスタンスを起動するとき、アグリゲートのデフォルトのディスクサイズを選択する必要があります。 Cloud Managerは、このディスクサイズを初期アグリゲートと、シンプルプロビジョニングオプションを使用するときに作成する追加アグリゲートに使用します。 拡張割り当てオプションを使用すると 、デフォルトとは異なるディスクサイズを使用するアグリゲートを作成できます 。

    アグリゲート内のすべてのディスクは同じサイズでなければなりません。

    ディスクサイズを選択するときは、いくつかの要因を考慮する必要があります。ディスクサイズは、ストレージに対する支払い額、アグリゲートで作成できるボリュームのサイズ、Cloud Volumes ONTAPで使用可能な合計容量、およびストレージパフォーマンスに影響を与えます。

    Azure Premium Storageのパフォーマンスは、ディスクサイズに関係しています。より大きなディスクは、より高いIOPSとスループットを提供します。たとえば、1 TBのディスクを選択すると、500 GBのディスクよりも高いコストでパフォーマンスが向上します。

    標準ストレージのディスクサイズにパフォーマンスの違いはありません。必要な容量に基づいてディスクサイズを選択する必要があります。

    ディスクサイズごとのIOPSとスループットについては、Azureを参照してください。

    • Microsoft Azure：マネージドディスクの価格

    • Microsoft Azure：ページブロブの価格

    GCPでシステムのサイズを決定する

    Cloud Volumes ONTAPシステムのサイジングは、パフォーマンスと容量の要件を満たすのに役立ちます。マシンの種類、ディスクの種類、およびディスクサイズを選択する際には、いくつかの重要な点に注意する必要があります。

    マシンタイプ

    Cloud Volumes ONTAPリリースノートでサポートされているマシンタイプを確認し、サポートされている各マシンタイプに関するGoogleからの詳細を確認します。ワークロード要件を、マシンタイプのvCPUとメモリの数に一致させます。各CPUコアがネットワークパフォーマンスを向上させることに注意してください。

    詳細については、次を参照してください。

    • Google Cloudドキュメント：N1標準マシンタイプ

    • Google Cloudドキュメント：パフォーマンス

    GCPディスクタイプ

    Cloud Volumes ONTAPのボリュームを作成する場合、Cloud Volumes ONTAPがディスクに使用する基盤となるクラウドストレージを選択する必要があります。ディスクタイプは、 Zonal SSD永続ディスクまたはZonal標準永続ディスクのいずれかです。

    SSD永続ディスクは、ランダムIOPSの高いレートを必要とするワークロードに最適ですが、標準永続ディスクは経済的で、順次読み取り/書き込み操作を処理できます。詳細については、 Google Cloudのドキュメント：Zonal Persistent disks（Standard and SSD）をご覧ください。

    GCPディスクサイズ

    Cloud Volumes ONTAPシステムを展開するときは、初期ディスクサイズを選択する必要があります。その後、Cloud Managerにシステムの容量を管理させることができますが、自分でアグリゲートを構築する場合は、次のことに注意してください。

    • アグリゲート内のすべてのディスクは同じサイズでなければなりません。

    • パフォーマンスを考慮しながら、必要なスペースを決定します。

    • 永続ディスクのパフォーマンスは、ディスクサイズとシステムで使用可能なvCPUの数に応じて自動的にスケーリングされます。

      詳細については、次を参照してください。

      • Google Cloudのドキュメント：Zonal Persistent Disk（StandardおよびSSD）

      • Google Cloudドキュメント：永続ディスクとローカルSSDのパフォーマンスの最適化
